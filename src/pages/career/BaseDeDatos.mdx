## Base De Datos

Despues de las hermosas clases de Arquitectura y Sistemas Operativos , volvimos a las clases de Rosalía con Base de datos , donde la primera clase ya fue una clase compleja de crear diagramas entidad Relación y una entrega para el fin de semana

### Clase I - 16 de agosto de 2024

<div class="w-full md:w-[800px] min-h-80 border bg-white shadow-lg p-4 md:p-6 mx-auto my-4 rounded-lg">
  <h3 class="text-3xl font-bold text-center mb-6 text-blue-600">Introducción</h3>
  <p class="text-lg text-gray-700 mb-4">
    En el entorno del mercado actual, la competitividad y la rapidez de maniobra de una empresa son imprescindibles para su éxito. 
    Para conseguirlo existe <span class="font-semibold">cada vez una mayor demanda de datos</span> y, por tanto, 
    <span class="font-semibold">más necesidad de gestionarlos</span>. Esta demanda siempre ha estado patente en empresas y sociedades, 
    pero en estos años se ha disparado debido al acceso multitudinario a las redes integradas en Internet y a la aparición de los 
    dispositivos móviles que también requieren esa información.
  </p>
  <p class="text-lg text-gray-700 mb-4">
    En informática se conoce como <span class="font-semibold">dato</span> a cualquier elemento informativo que tenga 
    <span class="font-semibold">relevancia para un usuario</span>. Desde su nacimiento, la informática se ha encargado de proporcionar 
    herramientas que faciliten la manipulación de los datos.
  </p>
  <p class="text-lg text-gray-700">
    Los sistemas de información actuales se basan en <span class="font-semibold">bases de datos (BD)</span> y 
    <span class="font-semibold">sistemas de bases de datos (SGBD)</span> que se han convertido en elementos imprescindibles de la 
    vida cotidiana de la sociedad moderna.
  </p>
</div>

<div class="w-full md:w-[800px] min-h-80 border bg-white shadow-lg p-6 mx-auto my-4 rounded-lg">
  <h3 class="text-3xl font-bold text-center mb-6 text-blue-600">Definición de Base de Datos</h3>
  <p class="text-lg text-gray-700 mb-4">
    Cada día, la mayoría de nosotros nos encontramos con actividades que requieren algún tipo de interacción con una base de datos 
    (ingreso en un banco, reserva de una entrada para el teatro, solicitud de una suscripción a una revista, compra de productos, ...).
  </p>
  <ul class="list-disc list-inside text-gray-700 mb-4">
    <li><span class="font-semibold">Una base de datos</span> se entenderá como una colección de datos relacionados entre sí y que tienen un significado implícito.</li>
    <li><span class="font-semibold">Por datos</span> queremos decir hechos conocidos que pueden registrarse y que tienen un significado implícito.</li>
  </ul>
  <p class="text-lg text-gray-700 mb-4">
    <span class="font-semibold">Ejemplo:</span> Una agenda con los nombres y teléfonos de un conjunto de personas conocidas es una base de datos, 
    puesto que es una colección de datos relacionados con un significado implícito.
  </p>
  <p class="text-lg text-gray-700 mb-4">
    La definición hace referencia a dos elementos para que un conjunto de datos constituya una Base de Datos:
  </p>
  <ol class="list-decimal list-inside text-gray-700 mb-4">
    <li><span class="font-semibold">Relaciones entre datos</span>, tema que se tratará en las secciones siguientes.</li>
    <li><span class="font-semibold">Significado implícito de los datos</span> que se atribuye dependiendo del contexto en que se utilizan los mismos.</li>
  </ol>
</div>

<div class="w-full md:w-[800px] min-h-80 border bg-white shadow-lg p-6 mx-auto my-4 rounded-lg">
  <h3 class="text-3xl font-bold text-center mb-6 text-blue-600">Evolución Histórica</h3>
  <p class="text-lg text-gray-700 mb-4">
    Los predecesores de los sistemas gestores de bases de datos fueron los sistemas gestores de ficheros o sistemas de archivos tradicionales.
  </p>
  <ol class="list-decimal list-inside text-gray-700 mb-4">
    <li><span class="font-semibold">Archivos tradicionales:</span> Consiste en almacenar los datos en archivos individuales, exclusivos para cada 
      <span class="font-semibold">aplicación particular</span>. En este sistema los datos pueden ser <span class="font-semibold">redundantes</span> 
      (repetidos innecesariamente) y la <span class="font-semibold">actualización</span> de los archivos es más lenta que en una base de datos.</li>
    <li><span class="font-semibold">Base de datos:</span> Es un <span class="font-semibold">almacenamiento</span> de datos formalmente definido, 
      <span class="font-semibold">controlado centralmente</span> para intentar servir a múltiples y diferentes aplicaciones. La base de datos es 
      una fuente de datos que son compartidos por numerosos usuarios para diversas aplicaciones.</li>
  </ol>
</div>

<div class="w-full md:w-[800px] min-h-80 border bg-white shadow-lg p-6 mx-auto my-4 rounded-lg">
  <h3 class="text-2xl font-bold text-center mb-4 text-blue-600">Evolución de los Sistemas de Base de Datos</h3>
  <p class="text-lg text-gray-700 mb-4">
    Inicio: parte de la necesidad de almacenar grandes cantidades de datos.
  </p>
  <ol class="list-decimal list-inside text-gray-700 mb-4">
    <li><span class="font-semibold">Década de 1950:</span> Da origen a las cintas magnéticas.</li>
    <li><span class="font-semibold">Década de 1960:</span> Implementación de los discos.</li>
    <li><span class="font-semibold">Década de 1970:</span> Nacen las bases de datos relacionales y Oracle.</li>
    <li><span class="font-semibold">Década de 1980:</span> El modelo relacional se posiciona en el mercado de las bases de datos.</li>
    <li><span class="font-semibold">Actualidad:</span> Diversidad de modelos y tecnologías.</li>
  </ol>
  <h4 class="text-3xl font-bold text-center mb-6 text-blue-600">Evolución de los Tipos de Base de Datos</h4>
  <p class="text-lg text-gray-700 mb-4">
    Concidiendo con la evolución histórica de las bases de datos, estas han utilizado distintos modelos:
  </p>
  <ul class="list-disc list-inside text-gray-700 mb-4">
    <li>Jerárquicos</li>
    <li>En red</li>
    <li>Relacionales <span class="font-semibold">(Lo que vamos a estudiar)</span></li>
    <li>Multidimensionales</li>
    <li>De objetos</li>
  </ul>


  <h4 class="text-2xl font-bold mb-6 text-blue-400 text-center">Base De Datos Estructura Jerárquica</h4>
  
  <p class="text-lg text-gray-700 mb-4">
    La estructura jerárquica fue usada en las primeras bases de datos. Las relaciones entre registros forman una estructura en árbol. Actualmente, las bases de datos jerárquicas más utilizadas son IMS de IBM y el Registro de Windows en Microsoft.
  </p>

  <h4 class="text-2xl font-bold mb-6 text-blue-400 text-center">Base De Datos Con estructura en red</h4>

  <p class="text-lg text-gray-700 mb-4">
    Esta estructura contiene relaciones más complejas que las jerárquicas. Admite relaciones de cada registro con varios que se pueden seguir por distintos caminos.
  </p>

  <img src="/BaseDeDatosEstructuraDeRed.avif" alt="Base De Datos Estructura De Red" title="Imagen de Una Estructura De red" class="max-w-80 md:max-w-[500px] mx-auto" />

  <h4 class="text-2xl font-bold mb-6 text-blue-400 text-center">Base De Datos Con estructura relacional</h4>

  <p class="text-lg text-gray-700 mb-4">
    La <span class="font-semibold">Estructura Relacional</span> es la  más extendida hoy en día. Almacena los datos en <span class="font-semibold">Filas</span> o <span class="font-semibold">Registros</span>  (<span class="font-semibold text-green-600 uppercase">Tuplas</span>) y <span class="font-semibold">Columnas</span> o <span class="font-semibold">Campos</span>  (<span class="font-semibold text-green-600 uppercase">Atributos</span>). Estas tablas pueden estar conectadas entre sí por claves comunes.
  </p>

  <h4 class="text-2xl font-bold mb-6 text-blue-400 text-center">Base De Datos Con estructura orientada a Objetos</h4>

  <p class="text-lg text-gray-700 mb-4">
    La <span class="font-semibold">Estructura Orientada a Objetos</span> está diseñada siguiendo el paradigma de los lenguajes orientados a objetos. De este modo soporta los tipos de datos <span class="font-semibold">Graficos</span>, <span class="font-semibold">Imágenes</span>, <span class="font-semibold">Voz</span> y <span class="font-semibold">Texto</span> de manera natural. Esta Estructura tiene gran difusión en aplicaciones web para aplicaciones multimedia.
  </p>

  <figure class="max-w-80 md:max-w-[500px] mx-auto">

  <h3 class="text-2xl font-bold mb-4 text-blue-600">Sistemas de ficheros tradicionales</h3>
  
  <p class="text-lg text-gray-700 mb-4">
    En estos sistemas, cada programa almacenaba y utilizaba sus propios datos de forma un tanto caótica. La única ventaja que conlleva esto es que los procesos son independientes, por lo que la modificación de uno no afecta al resto.
  </p>
  
  <h4 class="text-xl font-semibold text-gray-800 mb-2">Inconvenientes:</h4>
  <ul class="list-disc list-inside text-gray-700">
    <li>Datos redundantes. Ya que se repiten continuamente.</li>
    <li>Coste de almacenamiento elevado. Al almacenarse varias veces el mismo dato en distintas aplicaciones, se requiere más espacio en los discos.</li>
    <li>Tiempos de procesamiento elevados. Al no poder optimizar el espacio de almacenamiento.</li>
    <li>Probabilidad alta de inconsistencia en los datos. Ya que un proceso cambia sus datos y no el resto. Por lo que el mismo dato puede tener valores distintos según qué aplicación acceda a él.</li>
    <li>Difícil modificación en los datos. Debido a la probabilidad de inconsistencia, cada modificación se debe repetir en todas las copias del dato (algo que normalmente es imposible).</li>
  </ul>

  <figure class="max-w-80 md:max-w-[500px] mx-auto">

  <h3 class="text-2xl font-bold text-gray-800 mb-4">Sistemas de bases de datos relacionales</h3>

  <h4 class="text-xl font-semibold text-gray-800 mb-2">Ventajas:</h4>
  <ul class="list-disc list-inside text-gray-700">
    <li>Menor redundancia. No hace falta tanta repetición de datos.</li>
    <li>Menor espacio de almacenamiento. Gracias a una mejor estructuración de los datos.</li>
    <li>Acceso a los datos más eficiente. La organización de los datos produce un resultado más óptimo en rendimiento.</li>
    <li>Datos más documentados. Gracias a los metadatos que permiten describir la información de la base de datos.</li>
    <li>Independencia de los datos y los programas y procesos. Esto permite modificar los datos sin modificar el código de las aplicaciones.</li>
    <li>Integridad de los datos. Mayor dificultad de perder los datos o de realizar incoherencias con ellos.</li>
    <li>Mayor seguridad en los datos. Al limitar el acceso a ciertos usuarios.</li>
  </ul>

  <h4 class="text-xl font-semibold text-gray-800 mt-4 mb-2">Desventajas:</h4>
  <ul class="list-disc list-inside text-gray-700">
    <li>Instalación costosa. El control y administración de bases de datos requiere de un software y hardware potente.</li>
    <li>Requiere personal cualificado. Debido a la dificultad de manejo de este tipo de sistemas.</li>
    <li>Implantación larga y difícil. Debido a los puntos anteriores, la adaptación del personal es mucho más complicada y lleva bastante tiempo.</li>
  </ul>

  <h3 class="text-2xl font-bold text-gray-800 mb-4">Diseño de Base de Datos</h3>

  <div class="bg-blue-50 border-l-4 border-blue-500 text-blue-700 p-4 mb-4">
    <h4 class="text-xl font-semibold">Requerimientos de la Información</h4>
    <p class="mt-2">
    Esta etapa consiste en identificar y recopilar toda la información necesaria para el sistema de base de datos. Implica entender qué datos se deben almacenar, cómo se relacionan entre sí, y qué consultas o informes serán necesarios. Es fundamental para garantizar que la base de datos cumpla con las necesidades de los usuarios y el funcionamiento de la organización.
    </p>
  </div>

  <h4 class="text-xl font-semibold text-gray-800 mb-2">Etapas de diseño para la base de datos:</h4>

  <ul class="list-none text-gray-700">
    <li class="mb-2">
      <span class="text-green-600 font-bold">1. Diseño conceptual:</span> Definición de los requerimientos de información y estructura general de la base de datos.
    </li>
    <li class="mb-2">
    <span class="text-red-600 font-bold">2. Diseño lógico:</span> Definición de las tablas, relaciones, y reglas de integridad de acuerdo con el modelo conceptual.
    </li>
    <li class="mb-2">
    <span class="text-yellow-500 font-bold">3. Diseño físico de la base de datos:</span> Definición de la estructura física, tipos de datos, índices y cómo se almacenará la información.
    </li>
  </ul>

  <h4 class="text-xl font-semibold text-gray-800 mt-4 mb-2">4. Implementación de la base de datos</h4>
  <p class="text-gray-700">
      La etapa final donde la base de datos es creada y puesta en funcionamiento basada en los diseños conceptual, lógico y físico.
  </p>

  <div class="bg-yellow-50 border-l-4 border-yellow-500 text-yellow-700 p-4 mb-4">
    <h4 class="text-xl font-semibold">Diseño de Bases de Datos</h4>
    <p class="mt-2">
      <strong>Fase de Análisis: Especificación de Requisitos Software (E.R.S.)</strong> Antes de pasar a diseñar una base de datos (BD), es crucial tener claro qué se quiere lograr. Para ello, los informáticos se reúnen con los futuros usuarios del sistema para recopilar la información que necesitan y entender las expectativas de dichos usuarios.
    </p>
    <p class="mt-2">
      Normalmente, se realiza una reunión inicial y, a partir de ella, se elabora una batería de preguntas para entrevistar a los usuarios finales en una segunda reunión. Esta segunda entrevista permite obtener información detallada sobre lo que se espera de la base de datos.
    </p>
    <p class="mt-2">
      De estas entrevistas, se extrae el documento más importante del análisis, el <strong>documento de Especificación de Requisitos Software (E.R.S.)</strong>. A partir de este documento, se obtiene toda la información necesaria para la modelización de datos.
    </p>
  </div>
  
  <h5 class="font-semibold">Fase 1 del Diseño: Diseño Conceptual - Modelo Entidad/Relación (E/R)</h5>
  <p>
    El objetivo de esta fase es representar la información obtenida del usuario final y concretada en el E.R.S., utilizando estándares para que el resto de la comunidad informática pueda entender el modelo. El modelo que se utiliza en esta primera fase tiene un gran poder expresivo para comunicarse con usuarios no expertos en informática y se denomina <strong>Modelo Conceptual</strong>. En esta fase, empleamos el <strong>Modelo Entidad/Relación</strong>.
  </p>

  <h5 class="font-semibold">Fase 2 del Diseño: Diseño Lógico - Modelo Relacional</h5>
  <p>
    Este modelo es más técnico y está orientado al personal informático, con traducción directa al modelo físico que entiende el SGBD. Se obtiene a partir del modelo conceptual y dependerá de la implementación de la base de datos. La implementación varía según el tipo de BD: jerárquica, orientada a objetos o relacional.
  </p>
  <p class="font-semibold">Ejemplos:</p>
  <p>Productos, Inventario, Inventario, Categorías, Relaciones entre estas entidades que reflejan cómo los datos se interconectan y se organizan.</p>

  <h5 class="font-semibold">Fase 3 del Diseño: Diseño Físico - Modelo Físico</h5>
  <p>
    Es el resultado de aplicar el modelo lógico a un SGBD específico. Generalmente, se expresa en un lenguaje de programación de bases de datos como SQL. En esta etapa, transformaremos el <strong>Modelo Relacional</strong> en el modelo físico utilizando el sublenguaje DDL de SQL.
  </p>

</div>

### Clase I - Parte 2 - 16 de agosto de 2024

<div class="w-full md:w-[800px] min-h-80 border bg-white shadow-lg p-4  mx-auto my-4 rounded-lg">
  <h3 class="text-3xl font-bold text-center mb-6 text-blue-600">Modelo Entidad-Relación</h3>
  <p>
    El <span class="font-semibold">modelo Entidad-Relación </span>
    es el modelo más utilizado para el diseño conceptual de bases de datos. Fue introducido por Peter Chen en 1976 y se basa en la existencia de objetos a los que se les da el nombre de entidades, y asociaciones entre ellos, llamadas relaciones.
  </p>
  <img src="/PeterCheen.avif" alt="Introductor de El modelo Entidad Relación" title="Imagen de Peter Cheen" class="max-w-80 md:max-w-[500px] mx-auto" />
  <figcaption class="text-center text-gray-600 italic">Creado por Peter Chen en 1976</figcaption>

  <p>Sus símbolos principales se representan en el cuadro siguiente:</p>
  <img src="/BaseDeDatosSimbolosEntidadRelacion.avif" alt="Simbolos Entidad Relación" title="Imagen de Simbolos Entidad Relación" class="max-w-80 md:max-w-[500px] mx-auto" />

  <h4 class="text-xl font-semibold">Entidades</h4>
  <p>
    Una <strong>entidad</strong> es cualquier objeto o elemento acerca del cual se puede almacenar información en la <strong>base de datos</strong>. Las entidades pueden ser <strong>concretas</strong>, como personas y vehículos, o <strong>abstractas</strong>, como materias en una institución o cuentas bancarias.
  </p>
  <p>
    Gráficamente, las entidades se representan mediante <strong>rectángulos</strong>, con su nombre escrito en el interior. Es importante destacar que un nombre de entidad solo puede aparecer una vez en el <strong>esquema conceptual</strong>, asegurando la claridad y la unicidad de la información.
  </p>
  <p>
    Existen dos tipos de entidades: las entidades <strong>fuertes</strong>, que pueden existir de manera independiente, y las entidades <strong>débil</strong>, cuya existencia depende de la existencia de otra entidad. Este concepto es crucial para entender la <strong>estructura</strong> y las <strong>relaciones</strong> dentro de una base de datos.
  </p>

  <h4 class="text-xl font-semibold">Atributos</h4>
  <p>
    Una entidad se caracteriza y distingue de otra por los <strong>atributos</strong>, también llamados propiedades o campos, que representan las características de una entidad.
  </p>
  <p>
    Los atributos de una entidad pueden tomar un conjunto de valores permitidos al que se le conoce como <strong>dominio del atributo</strong>. Dando valores a estos atributos, se obtienen las diferentes ocurrencias de una entidad.
  </p>

   <h5 class="font-semibold mt-4">Tipos de Atributos:</h5>

  
<div class="flex flex-col space-y-4">
  <div>
    <p class="font-semibold underline">Identificadores:</p>
    <p>Definición: Son atributos cuyos valores no se repiten dentro de una misma entidad o relación. Actúan como clave principal o primaria.</p>
    <div class="border-4 border-black rounded-full flex items-center justify-center px-4 max-w-40">Identificadores</div>
    <p>Ej: dni</p>
  </div>
  
  <div>
    <p class="font-semibold">Descriptores:</p>
    <p>Definición: Atributos que describen diversas propiedades de una entidad o relación. Son los más frecuentes.</p>
    <div class="border-4 border-black rounded-full flex items-center justify-center px-4 max-w-40">Descriptores</div>
    <p>Ej: nombre</p>
  </div>

  <div>
    <p class="font-semibold">Derivados:</p>
    <p>Definición: Atributos cuyos valores se calculan a partir de los valores de otros atributos.</p>
    <div class="border-4 border-black border-dashed rounded-full flex items-center justify-center px-4 max-w-40">Derivados</div>
    <p>Ej: edad</p>
  </div>

  <div>
    <p class="font-semibold">Multivaluados:</p>
    <p>Definición: Atributos descriptores que poseen varios valores de un mismo dominio, utilizados para almacenar múltiples datos.</p>
    <div class="border-4 border-black border-double rounded-full flex items-center justify-center px-4 max-w-40">Multivaluados</div>
    <p>Ej: teléfono</p>
  </div>

  <div>
    <p class="font-semibold">Compuestos:</p>
    <p>Definición: Atributos que pueden ser descompuestos en otros atributos pertenecientes a distintos dominios.</p>
    <div class="border-4 border-black rounded-full flex items-center justify-center px-4 max-w-40">Compuestos</div>
    <p>Ej: fecha (Día, Mes, Año)</p>
  </div>
</div>


</div>
